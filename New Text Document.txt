## 🐛 WHAT DOESN'T MAKE SENSE / BUGS

### 🔴 Critical Logic Issues

**1. Death is consequence-free**
```
// Player dies → teleported to start with 50% HP
// No item loss, no XP loss, no penalty
player.hp = Math.floor(player.maxHp * .5);
player.x  = startPos.x;
```
**Problem:** Encourages reckless play. No consequence system exists.

---

**2. `isWalk()` uses floating point comparison**
```javascript
function monAt(x, y) {
  const ix = Math.floor(x), iy = Math.floor(y);
  return ents.find(e => e.x === ix && e.y === iy ...
}
```
But `isWalk()` uses `player.x === x` directly. Monsters and players always have integer coords — but passing floats anywhere would silently break collision. Should enforce integer-only positions.

---

**3. Monster push logic is inconsistently balanced**
```javascript
if (mon.maxHp < player.maxHp * .55) {
  // push monster
}
```
**Problem:** A level 1 player with 200 HP can push a Ghost (96 HP) because 96 < 110. This threshold doesn't scale meaningfully and lets you cheese monsters.

---

**4. Auto-attack fires from the game ticker (setInterval 200ms) AND from click/keydown**
```javascript
// setInterval (always running)
if (tgt && now - lastAutoAtk >= getAutoAttackCd() ...)

// Also fires in click handler
if (chebyshev(...) <= 1 && now - lastAutoAtk >= getAutoAttackCd())
  playerAtk(mon);

// Also fires in movePlayer
if (now - lastAutoAtk >= getAutoAttackCd())
  playerAtk(mon);
```
**Problem:** Triple-trigger risk. Can cause double-attacks on the same tick if all three paths evaluate simultaneously.

---

**5. `addInv` doesn't enforce the 16-slot cap**
```javascript
function addInv(id, qty) {
  const ex = player.inv.find(s => s.id === id);
  if (ex) ex.qty += qty;
  else    player.inv.push({ id, qty }); // ← no slot limit check
}
```
**Problem:** You can stack 50 different item types. UI only shows 16, but data has no cap. Items past slot 16 are invisible and inaccessible.

---

**6. Corpse loot window has no sanity check after auto-loot**
```javascript
function openCorpse(cor) {
  autoLootCorpse(cor);
  if (!cor.loot.length) { addLog('Corpse fully auto-looted.'); return; }
  lootWin = { cor, open:true };
```
If auto-loot is mid-run and empties a corpse while lootWin is open, `renderLootUI` handles it correctly — but `lootWin.cor` still holds a reference to a decayed entity that may have been removed from `ents`. Should add a guard.

---

**7. `getZoneLevel()` returns 0 for corridors**
```javascript
// If player is in a corridor (between rooms), no room matches → zone = 0
// This incorrectly labels deadly deep corridors as "SAFE AREA"
```

---

**8. HP/MP regen won't trigger during the first 5 seconds of the game**
```javascript
let lastDmgTime = -99999; // ✅ This is fine, it means regen starts immediately
```
Actually this is OK — but `regenHpAcc` ticks only via `dt` in the game loop, while the 200ms setInterval also calls `updateUI`. These aren't connected — regen is fine but `updateUI()` in the interval is wasteful (it's also called every frame in `render()`).

---

### 🟡 Design / Balance Issues

**9. Skill "Swiftblade" (`baseAtkSpeed +15/level`) is effectively useless**
```javascript
function getAutoAttackCd() {
  return Math.max(260, AUTO_ATK_BASE - player.atkSpeed * 10);
}
// Max speed cap: 260ms = ~3.8 attacks/sec
// Swiftblade max (6 levels × 15 = 90 rating) → 1000 - 900 = 100ms → capped at 260ms anyway
// You'd hit the cap long before max Swiftblade
```
Dagger also gives +15 atkSpd. One dagger + 1 Swiftblade level = same cap as 6 Swiftblade levels.

---

**10. Resistances cap at 50% but nothing explains this in-game**
The cap is hardcoded but never communicated to the player. A player stacking fire resistance via `fireCloak` + `iceRing` (resCold:15) won't understand why it stops working.

---

**11. Water tiles are impassable but have no gameplay purpose**
Water exists visually but you can't cross it, fish in it, drown in it, or anything. It just blocks paths. Either needs mechanics or should be removed from the generator.

---

**12. `earthBelt` has `resEarth:12` but no monster deals Earth damage**
No monster in `MDEF` uses Earth damage type. The resistance is stat-bloat with zero function.

---

**13. `chaosRing` (`resChaos:12`) — same problem**
No monster deals Chaos damage.

---

**14. Shop has no stock limit / economy sink**
Player can buy infinite potions. No gold sinks beyond gear. Late game gold becomes meaningless.

---

**15. Level scaling is linear and breaks early**
```javascript
// Level up XP requirement
while (player.xp >= player.lv * 100)
// Level 1: 100 XP | Level 10: 1000 XP | Level 20: 2000 XP
// Rats give 12 XP — killing 8 rats = Level 2
// But monsters don't scale with player level
```
A level 10 player kills rats as easily as level 1. No difficulty curve beyond zone depth.

## 🏗️ ARCHITECTURAL NOTES

**What to refactor before scaling:**

1. **Entity system** — `ents[]` is a flat array searched with `.find()` everywhere. At 50+ entities this is O(n) per lookup. Replace with spatial hash or typed sub-arrays (`mons[]`, `items[]`, `cors[]`)

2. **`computeStats()` is called too often** — Called on every equip/unequip but also should be called on skill purchase. Currently it IS called there. Fine for now, but if adding 20+ stat sources, cache invalidation needs a dirty flag.

3. **`setInterval` + `requestAnimationFrame` overlap** — The 200ms interval updates UI and triggers auto-attacks. The rAF loop also calls `updateUI()` every frame via `render()`. Consolidate everything into the rAF loop, use delta time for the 200ms checks.

4. **`ITEM` and `MDEF` as flat objects** — Works now. Before adding 100+ items, move to a data file / JSON with a loader, and add item categories/tiers as first-class fields.
